# Knowledge v2: Facts Lockfile + Attested Overlay

**Status: SHELVED (owner decision 2026-07-24) — kept as a decision record and fallback only.**
**The operative design is the one-file Knowledge Entry model:**
**`docs/knowledge-one-file-contract.md` (facts live inside the entry, no separate lockfile).**

Original status line: approved design, not yet implemented
Owner decisions: 2026-07-23 (full new model; committed facts; chat-approve governance)

## Motivation

Discovery on the schema-v3 claim/evidence store at target scale (~250 Flows, ~400 objects,
~8k fields, ~800 Apex classes) measured:

- Draft store ≈ **1.6x the source size** (~30–40k YAML files, 50–70 MB raw). A large Flow
  (85 KB XML) drafts ~56 KB of claim+evidence; the weight sits in the claim's embedded
  `facts` (e.g. `dataOperations`), while evidence is only ~1 KB.
- `knowledge_registry` loads **every** claim YAML per `query`/`search`
  (flat `glob("*.yaml")` + PyYAML `SafeLoader`): measured ~0.8 ms per small claim and
  ~50 ms per large one — tens of seconds to minutes per query at scale, worse on the
  team's Windows machines.
- Governance throughput: ~30k claims / 25 per approval batch ≈ **1200+ human clicks**,
  repeated after every collector wave (a global `COLLECTOR_VERSION` bump flips the whole
  store to `verified-stale`).

The root cause is architectural: the store wraps two different kinds of content in the same
heavy lifecycle.

1. **Derived facts** (`assurance: observed`, ~95 % of volume) are a deterministic function of
   `(force-app commit, collector version)`. For these, the collector *is* the evidence; a
   human approving a structural extraction adds ceremony, not epistemic value.
2. **Attested content** (`assurance: inferred` — AI-written descriptions, business meaning,
   judgment-bearing heuristics) is where human governance genuinely matters.

The store was still empty at decision time, so no migration is needed.

## Constraints

- No SQL databases of any kind (owner constraint, 2026-07-23). Files only; plain-text,
  git-diffable artifacts. JSON/JSONL parses natively in C — no YAML-at-scale penalty.
- Windows-first team; no PowerShell dependencies; existing safety-hook/chat-approve
  mechanics are the promotion gate (owner chose chat-approve over ADO PR policies).
- The schema-v3 claim/evidence/review lifecycle **remains in the package** for future
  live-org observations (where evidence is a real observation receipt, not a copy of
  source). It is no longer used for repository-derived knowledge.

## Layer 1 — Facts lockfile (derived, generated, never approved)

`.ai/knowledge/facts/<MetadataType>.jsonl` — committed, generated, CI-checked.

- First line: file header `{ "kind": "knowledge-facts-header", "metadataType": …,
  "repositoryCommit": …, "collectorVersion": …, "componentCount": …, "generatedAt": … }`.
- One JSON line per component: `componentId`, subject identity, `sourcePath`,
  `sourceDigest`, `facts` (exact output of the existing extractors, including the
  sanitizer), `references` (the existing kind vocabulary / usage registry), and
  per-component `limitations`.
- Granularity: one line per component as the extractors define components today
  (fields stay individual lines — line count is free in JSONL and preserves
  `--uses-field` precision).
- Regenerated by `facts build`; `facts build --check` fails CI when the committed
  lockfile does not byte-match a regeneration at HEAD (same contract as
  `render-indexes --check`). Validity is structural: commit + digests match, or the file
  is rebuilt. **Drift and refresh waves cease to exist for facts** — a collector upgrade
  is one regeneration commit, not a re-approval campaign.
- Sanitization (`sanitize_literal`, credential/PII drops, host-only endpoints) stays in
  the collector and therefore applies before anything reaches the lockfile.

## Layer 2 — Overlay (attested, human-governed)

`.ai/knowledge/overlay/…` — markdown, one file per component (or per object for
object-scoped content), in the human-browsable structure:

```
overlay/
  objects/<Object>.md            object meaning + notable fields, VRs, workflow context
  flows/<Object>/<Flow>.md       record-triggered (grouping key: facts.start.object)
  flows/_screen/<Flow>.md        processType without a triggering object
  flows/_autolaunched/<Flow>.md
  apex/<Class>.md                no object folder — object links are heuristic
  access/, integrations/, …
```

Each file has YAML frontmatter: `subject`, `metadataType`, `sourceDigest`,
`repositoryCommit`, `status: proposed | approved`, `reviewedBy`, `reviewedAt`,
`reviewBy`, `sensitivity`, `mechanism: copilot-chat-confirmation`.

- Body layout: a generated facts summary between `<!-- BEGIN/END GENERATED FACTS -->`
  markers (rendered from the lockfile — this is the single "final knowledge file" a human
  reads), followed by the attested free-text section (what the component does, business
  meaning, gotchas) written by the agent from actual source and approved by a human.
- Lifecycle: the agent can only create/update files with `status: proposed`
  (`<AGENT_…>` sentinel rejection carries over). `overlay approve` follows the existing
  chat-approve pattern: the safety hook answers `ask`, the human confirms in chat, the
  reviewer identity comes from local `knowledge.chatReviewer`, and the command stamps
  `approved` + reviewer + recomputed digests atomically.
- Drift: a `status`/`coverage` report compares frontmatter `sourceDigest` against the
  current source digest (the existing coverage math) and reports
  `approved-drifted` — advisory, never auto-mutating, same posture as today.
  `reviewBy` expiry checks carry over unchanged.

## Layer 3 — Entry points (search, index, citations)

- **Search**: the BM25 implementation and `search_corpus` composition port as-is, but read
  facts JSONL (lazy, per-type shards) + overlay bodies instead of 30k claim YAMLs.
  Corpus assembly at target scale is a JSON parse of tens of MB — seconds, not minutes;
  no cache layer needed under the no-DB constraint.
- **Index**: one small generated `index.json` (types, counts, file map, overlay coverage
  summary) as the agent entry point. No megabyte-scale flat domain files; the per-domain
  Markdown indexes are superseded by the overlay tree.
- **Citations**: designs cite `(path, repositoryCommit, sourceDigest)` for facts and
  `(overlay path, revision commit)` for attested content. `verify-citations` becomes a
  git + lockfile check (`ok / missing / digest-mismatch / not-approved`).

## Command surface (new `scripts/knowledge_store.py`)

| Command | Role | Governance |
|---|---|---|
| `facts build [--metadata-type T]` | regenerate lockfile shards | free (deterministic) |
| `facts build --check` | CI drift gate | free |
| `overlay draft <component-or-object>` | scaffold from facts + agent description | agent, `proposed` only |
| `overlay approve --path P` | chat-approve promotion | safety hook `ask` → human |
| `status` / `coverage` | facts freshness, overlay approved/proposed/drifted/missing, document-next priority | read-only |
| `search` / `--uses-object` / `--uses-field` / `--related` | BM25 + usage registry | read-only |
| `verify-citations --envelope <path>` | citation validity | read-only |

## What is reused vs retired

Reused unchanged: every extractor and the kind vocabulary in `force_app_knowledge.py`,
`sanitize_literal`, the coverage/worklist math, BM25 + corpus composition, the
chat-approve/safety-hook mechanism, metadata preflight, batching skills (re-pointed at
overlay batches).

Retired for repository knowledge: per-claim/evidence/review YAML pairs, `propose`/
`approve-claim` for structural facts, refresh waves, the per-domain generated Markdown
indexes, `claims-index.json` (replaced by `index.json`).

Retained dormant: the full claims registry and lifecycle contract for future live-org
observation claims.

## Implementation stages

1. **Facts**: `facts build`/`--check`, JSONL schemas, header contract, wire extractors,
   `index.json`, unit tests.
2. **Search/status**: port BM25 + usage flags, coverage/status report, verify-citations.
3. **Overlay**: draft scaffolding, frontmatter schema, chat-approve command + safety-hook
   wiring, drift report, sentinel checks.
4. **Harness integration**: role-guard allowlists (mirror `build_parser` flags — known
   drift gotcha, pin with a guard↔parser contract test), prompts/skills re-pointing,
   docs, CI pins (`facts build --check`, schema validation, count pins).
