# Template: Feature Dossier

The feature dossier is a **generated** draft artifact, not a hand-authored document. The feature
documentor writes it to `output/feature-dossiers/<slug>.md` from the crawl boundary
(`.cache/knowledge-proposals/feature-<slug>.json`) and the feature-tagged draft claims. This file
records the expected shape so a reviewer knows what to expect; edit the generator
(`scripts/force_app_knowledge.py render_dossier`), not the output.

A dossier is never verified Knowledge. Every fact in it corresponds to a `proposed` claim carrying a
`feature` tag until a human review promotes it. Do not publish it to ADO or a production wiki.

## Sections

```markdown
# Feature Dossier — <Feature name>

Draft documentation generated by the feature documentor from source-format metadata at commit
`<short commit>`. Not verified Knowledge...

## Overview
- Anchor objects: `<Object__c>`, `<Object__c>`
- Crawl depth: <n>
- Objects in boundary: <count>
- Drafted proposed claims: <count>
- Hub stop-list (not expanded): <objects>        # only when a hub list was supplied
- Unresolved anchors: <objects>                   # only when an anchor was not a known object

## Relations
### Outbound (anchor references another object)
| From | Field | To | Type |               # Lookup / MasterDetail / Hierarchy / ExternalLookup ...

### Inbound (another object references the anchor)
| From | Field | To | Type |               # reverse child relationships

### Junction objects (many-to-many bridges)     # objects with two or more MasterDetail fields

## Automations                                   # Flow, ApexClass, ApexTrigger, ApprovalProcess
| Type | Name | Description |

## UI surfaces                                    # FlexiPage, ApexPage, LWC/Aura, Layout, CustomTab
| Type | Name | Description |

## Supporting components                          # permission sets, custom metadata, validation rules, ...
| Type | Name | Description |

## Limitations                                    # source-only derivation caveats + org reconcile status

## Source traceability
- Repository commit: `<40-char commit>`
- Source tree digest: `sha256:<digest>`
- Crawl generated at: <UTC timestamp>
```

## Notes

- The `Description` column is filled from each behavior-bearing component's `component-description`
  draft. Until the agent replaces the `<AGENT_...>` sentinel with a real, source-grounded
  description, the cell reads `_description pending_` — the same gate the propose step enforces.
- Object association for FlexiPages, cross-object formula fields, and roll-up summaries is not
  derivable from parsed references; those appear in the Limitations section, not silently omitted.
- Live-org reconciliation is optional and additive: when a reviewer runs
  `python scripts/salesforce_read.py retrieve` for the boundary artifacts, record disk-vs-org drift
  in the Limitations section before promoting any claim.
